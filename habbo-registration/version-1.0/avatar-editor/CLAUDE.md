# Avatar Editor v1.0 — Claude Code Instructions

## Rules

- **Always document all changes.** For every change: document the current behavior and rationale in source code comments, README.md, and CLAUDE.md. This applies to every task, not just large ones.
- **Do not use "Old/New/Previous/Changed behavior" comment patterns.** Write descriptive comments explaining what the code does and why. No historical changelog narration in source comments or documentation.
- **Do not use `while` statements — use bounded `for` loops instead.** `while` loops are not friendly to JIT optimization by the V8 compiler. Use `for` loops with explicit bounds.

## Architecture

### SolidJS DOM Rendering

SolidJS 1.9.11 component tree with JSX, DOM rendering, reactive signals, SVG feColorMatrix color tinting, pre-generated CSS classes for all styling, native DOM events. DOM `<img>` elements render vectorized SVGs at native display resolution — zooming, high-DPI, and scaling all preserve vector crispness.

### Data Loading

Data files (`figuredata.csv`, `symbols.csv`, `spriteoffsets.csv`, `figure_editor.xml`) stored in `src/data/`, embedded via Vite `?raw` imports. Parsed synchronously at module scope — data available immediately on import, no network requests. `loadLocalizationFromUrl(url)` available for runtime override via `HabboRegistrationConfig.localization_url`.

### CSS Architecture

Three-layer CSS architecture for host page isolation, with zero inline `style=""` attributes:

1. **CSS Modules** — every component has a co-located `.module.css` file. Class names are hashed with `ae-` prefix (`ae-[name]__[local]__[hash:base64:5]` in dev, `ae-` + 6-char SHA-256 base64url hash in prod via `createHashNamer`). No class name clashes with host page.
2. **Pre-generated CSS classes** — `src/data/atlas-sprites.css` (~4766 rules) generated by `scripts/optimize-assets.ts`. Global `ae-` prefixed class names for atlas regions, sprite offsets, preview offsets, color swatches, and layout positions. Imported by components that use atlas-backed rendering. Not a CSS module — class names are global (prefixed to avoid collisions). Layout values from `layout.csv` also exposed as CSS custom properties (`--ae-canvas-width`, etc.) via `virtual:layout-vars.css`.
3. **Scoped CSS reset** — `App.module.css` applies a targeted reset on the root container (`box-sizing: border-box`, `margin/padding/border: 0` on all children, `max-width/max-height: none` on images).

**CSS class categories in atlas-sprites.css**:

| Category | Prefix | Example | Properties |
|----------|--------|---------|------------|
| Atlas page | `ae-ap-N` | `ae-ap-0` | `background-image` |
| Region | `ae-r-KEY` | `ae-r-sprites-42` | `width`, `height`, `background-position` |
| Avatar offset | `ae-so-ID` | `ae-so-42` | `left`, `top` |
| Avatar offset flipped | `ae-sof-ID` | `ae-sof-42` | `left`, `top` |
| Preview offset | `ae-po-ID-TYPE` | `ae-po-42-hr` | `left`, `top` |

**CSS custom property mangling** (production only): `cssVarMangling` Vite plugin assigns short hash-based names to all `--ae-` layout properties via `createHashNamer('--ae-')`. Virtual module `virtual:layout-vars.css` provides `:root` with `--ae-` layout vars from `layout.csv`.

### Hybrid Raster/SVG Rendering

Avatar sprite components use a `<Show>` switch between raster and SVG rendering:
- **Raster** (before vectorization): `<div>` with CSS sprite classes — `ae-ap-N` for background-image, `ae-r-KEY` for width/height/background-position. Tinting via `ae-tr-HEX` class (SVG feColorMatrix filter).
- **SVG** (after vectorization): `<img src={blobUrl}>` with CSS classes for sizing/positioning. `src` is an HTML attribute, not a CSS style. Tinting via `ae-tv-HEX` class (SVG feColorMatrix filter reference).

Hybrid components: `AvatarDisplay.tsx`, `PreviewIcon.tsx`. Raster-only components: `SpriteElement.tsx` (UI assets, never vectorized), `LoadingScreen.tsx` (loading frames, never vectorized).

### Runtime SVG Vectorization

Two-phase rendering strategy with runtime SVG vectorization via `libdepixelize-wasm` (0.2.1):

1. **Immediate**: Load WebP atlas (raw pixel-art packed at 1x) — app is usable instantly with CSS sprite `<div>` elements
2. **Vectorization behind loading screen**: Loading screen stays visible until BOTH data loading AND vectorization complete (+ 3s minimum). `initSvgStore()` checks build-time hash (`__VECTORIZE_DB_HASH__`) against stored value — if valid, cached SVGs on disk are reused (skipping 30-60+ seconds). Otherwise: extract sprite pixels → sequential `depixelizeImage()` calls (single lazy worker via global WorkerPool) → store SVG data on disk (Cache API on secure origins, IndexedDB on HTTP/file://) → blob URLs created on demand via ref-counted sync → worker pool terminated after vectorization (skipped if everything was cached)

Only `sprites/` entries (~560 avatar body parts) are vectorized. `ui/` entries (~19 items) stay raster — the quality/cost tradeoff is not worthwhile for small UI elements. `frames/` entries stay raster (loading animation only).

### Key Rendering Pipeline

```
WebP atlas (1x) → pixel extraction (2D canvas) → depixelizeImage() sequential loop (single lazy worker)
  → SVG data stored on disk via Cache API or IndexedDB (~30-200 MB on disk, ~23 KB Set on V8 heap)
  → worker pool terminated (terminateWorkerPool()) → WASM module + thread freed (~20-35 MB)
  → components mount → addNeededKey(key) increments ref count, schedules syncSvgs()
  → queueMicrotask syncSvgs() reconciles neededKeys vs activeSvgs
  → reads from Cache API → creates blob URL → img.decode() preload (SVG parsed + cached by browser)
  → svgTriggers.dirty(key) fires per-key reactive signal
  → getSvgBlobUrl() reactive lookup via svgTriggers.track(key) — return blob URL if active, null otherwise
  → <Show when={svgUrl()}> switches between raster <div> and SVG <img> (paints immediately, no blank frame)
  → key no longer needed → removeNeededKey(key) → syncSvgs() → URL.revokeObjectURL() + dirty(key) → raster fallback
```

### Ref-Counted Blob URL Management

Blob URLs are managed via ref-counted `neededKeys` Map + `activeSvgs` Map + `TriggerCache` from `@solid-primitives/trigger`:

```
neededKeys: Map<string, number>       // key → refCount (updated by components)
activeSvgs: Map<string, string>       // key → blobUrl (managed by syncSvgs)
svgTriggers: TriggerCache<string>     // per-key reactive tracking (auto-cleaning)
```

**Component lifecycle**: `addNeededKey(key)` increments ref count, `removeNeededKey(key)` decrements (deletes at 0). Both schedule `syncSvgs()` via `queueMicrotask`. Components use `createEffect(on(() => key, ...))` to track key changes and `onCleanup()` to release the last held key.

**syncSvgs()**: Microtask-batched reconciliation. Unload phase: keys in `activeSvgs` not in `neededKeys` → `revokeObjectURL()`, delete, `dirty(key)`. Load phase: keys in `neededKeys` not in `activeSvgs` → `retrieveSvg()` from disk → `createObjectURL()` → `img.decode()` preload → add, `dirty(key)`. Re-checks `neededKeys.has(key)` after each `await` to handle keys removed during async loading.

**Cascade prevention**: `createEffect(on(() => key, ...))` only tracks sprite key, not per-key signals. Key unchanged → no re-request → stable.

**Per-key signals via TriggerCache**: `svgTriggers.track(key)` creates reactive dependency, `svgTriggers.dirty(key)` invalidates. Auto-cleans signal entries when no component tracks them. Only the 1 affected component re-renders on load/unload.

**Ref-counted lifecycle pattern** (used by AvatarDisplay and PreviewIcon):
```tsx
let prevKey: string | null = null;
createEffect(on(() => key, (key) => {
  if (prevKey) removeNeededKey(prevKey);
  prevKey = key;
  if (key) addNeededKey(key);
}));
onCleanup(() => { if (prevKey) { removeNeededKey(prevKey); prevKey = null; } });
```

### Color Tinting

Unified tint system applied via runtime CSS classes generated by `SvgTintFilters.tsx`:

- **Both raster `<div>` and vectorized `<img>`**: SVG `<feColorMatrix>` filters defined as inline `<filter>` elements in `SvgTintFilters.tsx`. Raster tinting via `ae-tr-HEX` CSS class, vector tinting via `ae-tv-HEX` CSS class — both apply `filter: url(#tint-HEX)`. `color-interpolation-filters="sRGB"` ensures correct color multiplication. CSS `filter` applies to the rendered element output only — transparent pixels stay transparent.

`SvgTintFilters.tsx` renders both the SVG `<filter>` definitions and a companion `<style>` element with the tint CSS classes. This is the one exception to "no dynamic CSS rules" — tint classes are generated at runtime from the active color set (saves bandwidth vs pre-generating 215 colors × 2 rules in atlas-sprites.css).

`ColorTint.ts` provides helpers: `getRasterTintClass(hex)` → `ae-tr-HEX`, `getVectorTintClass(hex)` → `ae-tv-HEX`. Returns empty string for white (no tinting needed).

### UI Asset Rendering

`SpriteElement.tsx` renders UI assets as raster-only `<div>` elements with CSS sprite classes. No SVG vectorization for UI elements — they stay as CSS sprites with `image-rendering: pixelated`.

### Image Rendering Quality

`image-rendering` based on asset type via CSS module classes:
- **Raster (WebP atlas CSS sprites)**: `.raster` class → `image-rendering: pixelated` — sharp pixel-art edges
- **SVG (blob URLs after vectorization)**: `.vector` class → `image-rendering: crisp-edges` — clean vector edges
- **Loading frames**: Always `.frame` class with `image-rendering: pixelated` (never vectorized)
Applied in: `SpriteElement.module.css` (raster only), `AvatarDisplay.module.css`, `PreviewIcon.module.css`, `LoadingScreen.module.css`.

### Global Click Throttle

All buttons participate in global throttle via `clickThrottle.ts` — they call `triggerThrottle()` on click and check `isThrottled()` before accepting clicks. ContinueButton and RandomizeButton keep their own longer cooldowns (CONTINUE_DELAY, RANDOMIZE_COOLDOWN). All buttons show disabled state via `.disabled` CSS class (opacity 0.5, cursor default) during the 1s global throttle.

### What Gets Vectorized

- `sprites/` entries (~560 items) — avatar body parts, vectorized via `vectorizeSprites()`
- `ui/` entries (~19 items) — buttons, arrows, radio buttons, backgrounds — stay raster (not vectorized)
- `frames/` entries — loading animation frames, stay raster (CSS sprites + CSS invert) — only shown during brief loading screen

**Important**: Always use sequential `depixelizeImage()` loop, never `depixelizeBatch()` — parallel processing causes OOM with ~580 sprites.

### Memory Management

Post-vectorization cleanup reduces browser memory by 56-357 MB:

1. **Disk-backed SVG storage with cache validation** (saves 30-200 MB): SVG strings stored on disk — Cache API (`caches.open('avatar-svgs')`) on secure origins (HTTPS, localhost), IndexedDB fallback on non-secure origins (HTTP, file://). `hasCacheApi` flag checked at module init (`typeof caches !== 'undefined'`). `storeSvg()`/`retrieveSvg()`/`clearSvgStore()` abstract the backend. `vectorizedKeys` Set (~23 KB) tracks which keys have been vectorized. `syncSvgs()` reads via `retrieveSvg()` on demand (~1-5ms). **Build-time hash validation**: `__VECTORIZE_DB_HASH__` (SHA-256 of libdepixelize-wasm version + WebP atlas files) stored alongside SVGs. On load, `initSvgStore()` compares hash — match → reuse cached SVGs; mismatch → clear store, re-vectorize. Supports graceful partial cache (browser closed mid-vectorization → next session skips cached items, vectorizes the rest).

2. **No raster URL cache** (saves 0-116 MB): Raster rendering uses CSS sprite classes on `<div>` elements — no data URL extraction or caching needed.

3. **Worker pool termination** (saves 20-35 MB): `terminateWorkerPool()` called after `vectorizeSprites()` completes, but only if vectorization actually ran this session. If everything was cached, no worker was ever created — `destroyPool()` skipped. Terminates WASM module + worker thread.

**Ref-counted blob URL lifecycle**: Components call `addNeededKey(key)` / `removeNeededKey(key)` to acquire/release blob URLs. `syncSvgs()` runs via `queueMicrotask` after all effects complete — reconciles `neededKeys` vs `activeSvgs`, reads from disk cache, creates blob URLs, preloads via `img.decode()`, fires per-key signals via `svgTriggers.dirty(key)`. The `decode()` call ensures the browser has parsed and cached the SVG bitmap before the `<Show>` swap — `<img>` paints immediately with no blank frame. Temporary `Image` element is GC'd after scope; decoded data stays in browser cache until `revokeObjectURL()`. On decode failure, blob URL is revoked (prevents leak) and the component stays raster.

### Atlas Assets

Atlas WebP files imported via `import.meta.glob('../../assets/atlas_*.webp', { query: '?url&no-inline' })` in `Atlas.ts` — emitted as separate content-hashed `.webp` files in `dist/`. `assetsPath` optional override in `HabboRegistrationConfig` for non-Vite host pages that serve atlas files from a custom location.

### Build Configuration

ESM format. Build config split across three files:
- `vite-utils.ts` — pure utilities: `createHashNamer`, `computeVectorizeDbHash`, `esbuildOptions`
- `vite-plugins.ts` — Vite plugin factories: `stripRawWhitespace`, `cssVarMangling`, `esbuildMinifyPlugin`, `compressionPlugins`, `copyExternalDeps(dev, deps)`
- `vite.config.ts` — config assembly (`createViteConfig`) + `defineConfig` default export

`external: ['libdepixelize-wasm']` in rollupOptions — bare specifier preserved in output `import('libdepixelize-wasm')`, host page resolves via import map or bundler config.

Production builds include:
- `vite-plugin-compression2` — pre-compressed `.gz`, `.br`, `.zst` files at build time
- `esbuildMinifyPlugin` — Vite plugin (`enforce: 'post'`, `generateBundle` hook) that runs esbuild `transform()` on each JS chunk. Required because Vite's `build.minify` has no effect on ES library output (vitejs/vite#6555)
- ESBuild options — `target: 'esnext'`, tree-shaking, `drop: ['console']` in prod
- Rollup compact output — `compact: true`, arrow functions, const bindings, object shorthand
- Rollup treeshake — `moduleSideEffects: false`, `propertyReadSideEffects: false`, `tryCatchDeoptimization: false`
- `stripRawWhitespace` plugin — production only, minifies embedded CSV/XML `?raw` strings
- `cssVarMangling` plugin — virtual module `virtual:layout-vars.css` (`:root` with `--ae-` layout vars from `layout.csv`). Production: mangles `--ae-` layout properties to short SHA-256 hash names via `createHashNamer`
- `createHashNamer(prefix)` — generates short, collision-free hash-based names (SHA-256 base64url, 6-char start). Used for both CSS module class names and CSS custom properties in production
- `css.modules.generateScopedName` — `ae-[name]__[local]__[hash:base64:5]` in dev, `createHashNamer('ae-')` function in prod
- `copyExternalDeps` plugin — minifies, mangles, tree-shakes, and compresses externalized deps (libdepixelize-wasm, comlink) into `dist/libs/`. esbuild `transform()` with full optimization (minify whitespace/syntax/identifiers, `treeShaking: true`, `drop: ['console']`, `legalComments: 'none'`) + manual zlib compression (gzip/brotli/zstd at max settings). Runs in `writeBundle` hook — external deps bypass the normal bundle pipeline
- `eslint.config.js` — flat config: `@eslint/js` recommended, `typescript-eslint` recommended, `eslint-plugin-solid` (SolidJS), `eslint-plugin-jsx-a11y` (accessibility)
